# Architecture Decision Records — Carrying Quota design-system foundation

This document records the load-bearing architecture decisions behind the
four-layer, platform-grade design-system foundation. Each ADR states the
context, the decision, and the consequences. They are the *why* behind the
code; `design-system.md` is the *how*.

---

## ADR-001 — Four strictly-separated layers, one-way dependency

**Context.** Carrying Quota is an editorial web application that must embed
interactive data figures inside long-form prose without the layout, type,
spacing, or rhythm system buckling. The previous foundation (5 layout
primitives + a union-type token module, with raw Tailwind arbitrary values
baked into class maps) was sound but not platform-grade.

**Decision.** Adopt four strictly-separated layers with a one-way dependency
direction:

```
Layer 0  @cq/tokens   token source of truth
   ↑
Layer 1  @cq/ui       token-typed UI primitives
   ↑
Layer 2  app shell    pure chrome (navbar / footer / outlet)
   ↑
Layer 3  routes + pieces   content only — zero hand-rolled geometry
```

No layer reaches "up." Tokens know nothing of primitives; primitives know
nothing of the shell; the shell knows nothing of route content.

**Consequences.** A change to a token propagates downward deterministically.
A route cannot smuggle geometry into the system because geometry only exists
as primitive props. The layers are independently testable. The cost is more
indirection than a single-package app — accepted, because the rigor is the
product: this foundation is the proof-of-quality that carries into the
AccountExecutive.com sales conversation.

## ADR-002 — One token source, three emitted artifacts

**Context.** Tokens were previously Tailwind class strings inside TypeScript
union types — three half-truths (the CSS, the Tailwind theme, the TS types)
that could drift.

**Decision.** `@cq/tokens` holds ONE structured TypeScript source
(`packages/tokens/src/*.ts`). A build pipeline (`scripts/build.mjs`) emits
THREE artifacts from it:

- `dist/tokens.css` — plain CSS custom properties (`--cq-*`)
- `dist/theme.css` — a Tailwind v4 `@theme` block
- `dist/index.{js,d.ts}` — the compiled, typed TS exports

The 8 token categories are spacing, type-scale, color, radius, shadow, motion
(duration + easing), breakpoint, and z-index.

**Consequences.** The three artifacts cannot drift — they are pure functions
of one source. The build is idempotent by construction: no timestamps, no
build ids, no absolute paths, stable key order — running it twice yields
byte-identical output. The CSS artifacts are generated by importing the
freshly-compiled `dist/index.js`, so the emitted CSS is provably the same
data as the typed exports.

## ADR-003 — The token color values are reverse-derived from the locked palette

**Context.** `src/styles/globals.css` carries the live palette: a `:root`
block of HSL triplets plus a hand-written `@theme inline` block. That file is
byte-locked for this cycle — the palette must not visibly change.

**Decision.** The `color` token category (`packages/tokens/src/color.ts`) is
reverse-derived, byte-for-byte, from the `globals.css` `:root` palette. Each
color token's leaf `.hsl` field is the exact HSL triplet from that block.
The generated `tokens.css` / `theme.css` therefore reproduce the palette
without altering a single value.

The live application keeps using the hand-written `@theme inline` block in
`globals.css`. The generated `theme.css` is an *additive* artifact — proof
that the token source can drive a Tailwind theme, and a loadable theme for
isolated environments (Storybook) — never a rewrite of `globals.css`.

**Consequences.** `globals.css` stays byte-identical (verified by sha256).
There is one palette, expressed twice (the locked CSS + the token source)
with a fidelity check (`check-platform-static.mjs t3`) asserting they agree.
A future palette change is a coordinated edit to both — deliberately, so the
lock is a real lock.

## ADR-004 — Token-typed primitive props; compile-time enforcement only

**Context.** The previous primitives accepted token *names* but also a raw
`className` passthrough, so any consumer could inject off-scale geometry.

**Decision.** Every geometry/scale prop on a `@cq/ui` primitive is typed to a
token union imported from `@cq/tokens` (`SpacingToken`, `ContainerWidth`,
`RadiusToken`, …). An off-scale value is a compile error. Enforcement is
purely compile-time — the TypeScript types. Runtime schema validation (zod
etc.) is deliberately NOT added: the compile-time guarantee is zero-cost,
strictly stronger (caught before ship, not at render), and adds no runtime
dependency.

**Consequences.** `<Stack gap="huge">` does not compile. The guarantee costs
nothing at runtime. The cost is that the token unions must be kept in sync
with the token source — they are, because they ARE the token source's
exported types.

## ADR-005 — No untyped `className`; one explicit escape hatch

**Context.** An untyped `className` passthrough is the hole through which a
design system leaks. But a fully-sealed primitive is impractical — there are
always one-off needs.

**Decision.** No primitive exposes a prop named `className`. The single
caller-facing class-injection point is an explicit `unsafe_className` prop
(`packages/ui/src/lib/props.ts`). The name is deliberately loud: it signals
an unchecked override. Internally, primitives compose literal class strings
via the `cn()` helper and set the DOM element's own `className` — that is an
implementation detail, not a public surface.

**Consequences.** A reviewer greps for `className` and finds only
`unsafe_className`. The escape hatch exists, so the system is practical, but
every use of it is visible and auditable. The `no-geometry` ESLint rule
exempts `unsafe_className` — using it is a conscious, reviewable choice.

## ADR-006 — The app shell is pure chrome

**Context.** If the shell (the element wrapping the routed `<Outlet/>`) sets
content geometry — a max-width, content padding, a gap — then content
geometry is split between the shell and the content, and neither owns it.

**Decision.** The app shell (`src/components/layout/Layout.tsx`) is pure
chrome: navbar, footer, the scroll container, the `<Outlet/>`. The `<main>`
element imposes zero content geometry — no finite `max-width`, no horizontal
padding, no flex/grid gap. Chrome utilities (`flex`, `min-h-screen`, a sticky
header) are fine. Content geometry is owned entirely by the `Container` /
`Section` primitives and the `Page` frame.

**Consequences.** There is exactly one owner of content measure and content
padding — the `Container` primitive. The shell can be restyled without
touching content layout, and vice versa. Verified at runtime by the S1 DOM
probe.

## ADR-007 — `Page` frame, with a `flush` mode for rhythm bands

**Context.** Every route needs a consistent frame and a single root marker.
But the homepage is a series of full-bleed `<section>` rhythm bands whose
alternating background colors must remain direct children of `<main>` (a
`main > section` selector drives the rhythm check). A wrapper `<div>` between
`<main>` and the sections would break that.

**Decision.** The `Page` primitive tags the route root with `data-page-root`.
It has two modes:

- default — renders a single `<div data-page-root>` wrapper (right for
  single-band routes);
- `flush` — renders a Fragment and stamps `data-page-root` onto its FIRST
  child element, introducing no wrapper. The homepage uses `flush` so its
  `<section>` bands stay direct children of `<main>`.

**Consequences.** Every route has exactly one `[data-page-root]`; the
homepage's vertical rhythm selector keeps matching. The `flush` stamping
relies on the first child forwarding `data-*` attributes to its rendered
element — the `Section` primitive does, by spreading rest props onto its
`<section>`.

## ADR-008 — The `Piece` / `Figure` editorial contract

**Context.** The Carrying Quota-specific requirement: an arbitrary
interactive data figure must drop into any editorial piece without breaking
the piece's vertical rhythm, and the prose around it must not be able to
hand-roll geometry around it. The specific essays and figures are not yet
known — the rigor must be earned generally.

**Decision.** Two editorial primitives carry the contract:

- `Piece` — the editorial container, the unit of enforcement (the analogue of
  `Page` for a route). It owns the reading measure and the vertical rhythm
  between editorial blocks. Tagged `data-piece-root`.
- `Figure` — wraps any embedded media (image OR interactive data figure). It
  owns the figure's framing, caption, and width mode (`inset` / `measure` /
  `wide` — a token choice, never a raw width). Tagged `data-figure`. A figure
  embedded via `Figure` is spaced by the `Piece`'s rhythm token like any
  other block; the `wide` break-out is a primitive mode, not a route-level
  negative margin.

**Consequences.** A data figure cannot break editorial rhythm, and prose
cannot hand-roll geometry around it — the embed is structurally constrained.
Proven by the demo piece (`DemoPiece`) embedding the reference figure
(`CompCurveFigure`).

## ADR-009 — The reference data figure is custom-built (SVG + React)

**Context.** Editorial pieces will embed interactive data experiences. A
charting library is the obvious reach.

**Decision.** Defer the charting-library decision. The reference interactive
figure (`CompCurveFigure`) is custom-built — SVG + React, every coordinate
computed from the data. It carries zero raw color or size: bar fills use
Tailwind token utility classes (`fill-accent`, `fill-card`), the only inline
geometry is SVG user-space numbers. No charting library enters
`dependencies`.

**Consequences.** No new app-level runtime dependency. The figure proves the
embed contract with a real, interactive, token-driven component. When the
real data experiences are scoped, a charting library can be evaluated against
concrete requirements rather than chosen speculatively now.

## ADR-010 — Geometry is un-bypassable on the content surface

**Context.** A design system that *can* be bypassed *will* be bypassed.

**Decision.** A custom ESLint rule, `cq/no-geometry`
(`eslint-rules/no-geometry.js`), bans raw geometry utilities (`mx-auto`,
`max-w-*`, `px-*`, `py-*`, `gap-*`, arbitrary `*-[…]`) on `className` in route
and editorial-piece files. It is wired into `npm run lint`. The
`unsafe_className` escape hatch is exempt — a conscious, visible override.

**Consequences.** Geometry on the content surface MUST flow through a
token-typed primitive prop; the linter makes it un-bypassable. The rule is
scoped to the content surface only — primitives and chrome are free to use
Tailwind utilities (they are the layer that *implements* geometry).

## ADR-011 — CI enforcement from day one

**Context.** A design system without enforcement decays.

**Decision.** Ship the enforcement harness with the foundation, not later:

- **Storybook** — one story per primitive (`*.stories.tsx` beside each
  primitive), so every primitive is documented and visually inspectable.
- **axe-core** — `npm run test:a11y` runs axe over the migrated routes + the
  demo piece; zero WCAG 2 A/AA violations is the gate.
- **Playwright visual regression** — `npm run test:visual` captures a
  per-route + per-piece screenshot baseline; drift fails the gate.
- **TypeScript-as-tokens** — the compile-time guarantee of ADR-004.

**Consequences.** Every primitive is documented; accessibility and visual
regressions are caught in CI; off-scale token values are caught by the
compiler. The foundation is not just built to a standard — it is held to one.

## ADR-012 — npm workspaces; the app resolves packages by source alias

**Context.** Extracting `@cq/tokens` and `@cq/ui` into packages requires a
monorepo layout. The repo uses npm + `package-lock.json`.

**Decision.** Use npm workspaces (`workspaces: ["packages/*"]`) — no package
manager switch. For Vite + Tailwind, the app resolves `@cq/ui` and
`@cq/tokens` via a Vite alias pointing at the package *source*
(`packages/ui/src/index.ts`), not the `node_modules` symlink. Two reasons:
(1) Tailwind v4's content scanner follows the Vite module graph but skips
`node_modules` — aliasing to the source keeps primitive class strings inside
the scanned tree, so primitive utilities are never purged; (2) `@cq/ui` needs
no build step — Vite consumes its TS source directly. TypeScript resolves the
packages via the `node_modules` workspace symlinks (the package `exports`
maps), so type-checking and bundling agree.

**Consequences.** One install links everything; Tailwind sees every primitive
class; `@cq/ui` ships as pure source with no build artifact to stale. The
`@cq/tokens` package does build (it must emit the three artifacts), and the
app's production build runs `build:tokens` first.
